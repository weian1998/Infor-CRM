Types de relation entre les tables
  • La table parent détermine le type de relation. Par exemple, lorsque la table CONTACT est la table parent, sa relation avec la table ACCOUNT est du type un-à-un. En revanche, si la table ACCOUNT est la table parent, sa relation est du type un-à-plusieurs. Dans la colonne Exemple, le nom des tables parent est en italique.
  • Vous ne pouvez pas utiliser directement des tables qui ont une relation plusieurs-à-plusieurs. Vous devez vous servir d'une table de liaison, c'est-à-dire d'une table intermédiaire dont le but est de résoudre les relations plusieurs-à-plusieurs en créant un lien entre les tables. Si vous avez besoin d'informations provenant des tables USERINFO et ACTIVITY, utilisez la table USER_ACTIVITY pour y accéder. La table USER_ACTIVITY est une table de liaison.

Pour définir le type de relation entre plusieurs tables, utilisez les termes suivants :

Terme Symbole Description Exemple
Un à un Chaque enregistrement unique d'une table est lié à un seul enregistrement d'une autre table. Supposons qu'un contact est associé à un seul compte. La relation entre les tables CONTACT et ACCOUNT est alors de type un-à-un (correspondance unique).
De zéro à un Chaque enregistrement unique d'une table est lié à un seul enregistrement d'une autre table ou à aucun enregistrement. Une relation zéro-à-un s'établit après la configuration d'une table avec une relation un-à-un avec une autre table ne contenant aucun enregistrement. Une activité peut n'être liée à aucune opportunité. Si elle est liée à une opportunité, elle n'est liée qu'à cette seule opportunité ; la relation entre les tables ACTIVITY et OPPORTUNITY est alors de type zéro-à-un.
Un à plusieurs Chaque enregistrement de la table est lié à un ou plusieurs enregistrements d'une autre table. Plusieurs contacts peuvent être associés à un compte, mais un compte au moins est toujours associé à un contact. La relation entre les tables ACCOUNT et CONTACT est alors de type un-à-plusieurs (correspondances multiples).
Zéro-à-plusieurs Chaque enregistrement unique d'une table peut être lié à aucun enregistrement ou à plusieurs enregistrements d'une autre table. Une relation zéro-à-plusieurs s'établit après la configuration d'une table avec une relation un-à-plusieurs avec une autre table ne contenant aucun enregistrement. Un contact peut n'avoir aucune activité ou en avoir plusieurs. La relation entre les tables CONTACT et ACTIVITY est alors de type zéro-à-plusieurs.
Plusieurs-à-plusieurs La table contient plusieurs enregistrements qui sont liés à plusieurs enregistrements d'une autre table. Un utilisateur peut avoir plusieurs activités et une activité peut être associée à plusieurs utilisateurs. La relation entre les tables USERINFO et ACTIVITY est alors de type plusieurs-à-plusieurs.
